엄격 모드(Strict mode)는 기본값이 `true`이며 다음 경로들을 구분합니다.

`/hello`
`/hello/`

`app.get('/hello')`는 `GET /hello/` 요청과 일치하지 않습니다.

엄격 모드를 `false`로 설정하면, 두 경로는 동일하게 취급됩니다.

```javascript
const app = new Hono({ strict: false })
```

---

**얘 뭐 하는 애냐?**
Hono의 엄격 모드(strict mode)는 라우팅할 때 URL 경로 끝에 붙는 슬래시(`/`)를 얼마나 깐깐하게 볼 거냐를 결정하는 설정입니다. 기본값은 `true`인데, 이건 "슬래시 하나라도 다르면 다른 길이야!"라고 인식하는 FM 스타일입니다. 반대로 `false`로 하면 "슬래시 있고 없고는 그냥 취향 차이 아니겠어? 대충 같은 걸로 치자" 하는 유연한 스타일이 되죠.

**왜 쓰는데?**
1.  **라우팅 일관성 및 예측 가능성 (strict: true)**: `/user`와 `/user/`를 명확히 다른 경로로 취급하면, 개발자가 의도한 대로 정확하게 라우팅 로직을 구성할 수 있습니다. 특히 REST API 설계에서 리소스 표현 방식에 대한 명확한 규칙을 적용하고 싶을 때 유용합니다. "우리 집 주소는 '번지'까지 정확해야 해!"라는 거죠.
2.  **사용자 편의성 및 SEO (strict: false)**: 사용자가 브라우저 주소창에 `/about`을 치든 `/about/`을 치든 같은 페이지를 보여주고 싶을 때 `strict: false`가 편합니다. 검색 엔진 최적화(SEO) 관점에서도 중복 콘텐츠 문제를 피하기 위해 하나의 표준 URL(canonical URL)로 통일하는 것이 좋은데, 이때 슬래시 유무를 너그럽게 처리해주면 도움이 될 수 있습니다. (물론, 서버 사이드 리다이렉션으로 확실하게 통일하는 게 더 정석이긴 합니다.) "손님이 길 약간 헷갈려도 대충 안내해드려" 느낌이죠.

**언제 적용되냐?**
Hono 인스턴스를 생성할 때 `new Hono({ strict: false })`처럼 옵션으로 설정합니다. 한번 설정되면 해당 Hono 앱의 모든 라우팅 규칙에 일괄적으로 적용됩니다. "우리 가게 방침은 오늘부터 이렇습니다!" 하고 선언하는 것과 같아요.

**쓸 때 꿀팁 및 주의사항:**
*   **기본값은 `true` (깐깐 모드)**: 아무 설정 안 하면 `/path`랑 `/path/`는 다른 놈으로 취급합니다. "어? 왜 내 라우트 안 먹지?" 싶을 때 슬래시 한번 확인해보세요. 은근히 자주 겪는 함정입니다.
*   **`strict: false` (널널 모드)의 편리함과 함정**:
    *   **편리함**: 사용자가 실수로 슬래시를 빼먹거나 더 넣어도 앱이 알아서 잘 찾아가니 사용자 경험이 부드러워질 수 있습니다.
    *   **함정**: 만약 `/resource`는 컬렉션을 보여주고, `/resource/` (또는 `/resource/:id`)는 특정 아이템을 보여주는 식으로 슬래시 유무에 의미를 부여해서 API를 설계했다면, `strict: false` 때문에 의도치 않게 두 경로가 합쳐져 버릴 수 있습니다. "좋은 게 좋은 거라지만, 선은 지키자"는 거죠.
*   **SEO 관점**: `strict: false`로 설정하더라도, 검색 엔진은 `/path`와 `/path/`를 다른 URL로 인식할 수 있습니다. 중복 콘텐츠로 인한 불이익을 피하려면, 한쪽을 정식 URL로 정하고 다른 쪽에서 301 리다이렉트를 해주는 것이 가장 확실한 방법입니다. Hono 미들웨어를 사용해서 이런 리다이렉션 로직을 추가할 수 있습니다. "교통정리는 확실하게!"
*   **일관성이 중요**: 프로젝트 시작할 때 `strict` 모드를 어떻게 할지 팀원들과 합의하고 쭉 밀고 나가는 게 좋습니다. 중간에 바꾸면 기존 라우트들이 예상과 다르게 동작할 수 있어서 혼란을 야기할 수 있습니다. "한번 정했으면 끝까지 간다!"
*   **다른 프레임워크와의 차이**: Express 같은 다른 프레임워크들은 이 슬래시 처리에 대해 각자 다른 기본 정책을 가지고 있거나, 더 세부적인 옵션(예: `express.Router({ strict: true, caseSensitive: false })`)을 제공하기도 합니다. Hono로 넘어오거나 함께 사용할 때 이런 차이점을 인지하고 있어야 합니다.
